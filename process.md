# Development Process

## Overview

Kiroku is a static GitHub Pages site. There is no staging server and no manual deployment step — merging to `main` is the deployment. Two CI workflows run automatically on every push.

---

## Repository structure (key files)

```
kiroku/
├── index.html                  # Homepage — all custom CSS lives inline here
├── about.html                  # About page — mirrors index.html interaction system
├── styles.css                  # Auto-generated by CI (do not edit by hand)
├── src/styles/tailwind-input.css  # Tailwind source file (directives only)
├── package.json                # npm scripts: build:css
├── testing.md                  # Lighthouse testing instructions
├── process.md                  # This file
└── .github/workflows/
    ├── build-assets.yml        # Compiles and commits styles.css
    └── validate-html.yml       # Lints all HTML files with htmlhint
```

---

## CI workflows

### 1. `build-assets.yml` — Tailwind CSS compilation

**Trigger:** Push or PR to `main`

**What it does:**
1. Checks out the repo
2. Installs npm dependencies (`npm ci`)
3. Runs `npm run build:css` — this calls Tailwind CLI v3 to scan `index.html` (and `kiroku-v2-test/preview.html`) for used class names, then outputs a minified `styles.css`
4. If `styles.css` has changed, commits it back to the branch with the message `chore(ci): build styles.css [skip ci]`
5. The `[skip ci]` tag prevents an infinite loop — the bot commit does not re-trigger the workflow

**Requires:** `permissions: contents: write` in the workflow file (so the bot can push back to the branch).

### 2. `validate-html.yml` — HTML linting

**Trigger:** Push to `main`, or any PR

**What it does:**
1. Checks out the repo
2. Installs Node.js 24
3. Finds all `*.html` files via `git ls-files`
4. Runs `npx htmlhint` on each file
5. Fails the workflow if any htmlhint error is found

**Local equivalent:**
```zsh
npx htmlhint index.html about.html
```

## Sequence of events: push to main

```
Developer pushes commit to main
        │
        ├─► build-assets.yml starts
        │       ├── npm ci
        │       ├── tailwindcss --minify → styles.css
        │       └── if styles.css changed:
        │               git commit "chore(ci): build styles.css [skip ci]"
        │               git push  ──► styles.css updated on main
        │
        ├─► validate-html.yml starts
        │       └── npx htmlhint *.html
        │               pass → ✅  /  fail → ❌ (blocks merge on PRs)
        │
        └─► GitHub Pages rebuild triggered
                └── site live at kiroku.io within ~60s
```

## Pre-push checklist

Before pushing or opening a PR, verify:

- [ ] Custom CSS changes are in the inline `<style>` block in `index.html` / `about.html` — **not** in `styles.css` (it gets overwritten by CI)
- [ ] Any new Tailwind utility classes are used directly in HTML (so the Tailwind scanner picks them up)
- [ ] HTML linter passes locally: `npx htmlhint index.html about.html`
- [ ] Lighthouse scores are all 100: see [testing.md](testing.md)
- [ ] Version number in both page footers is up to date
- [ ] Changes mirrored to `about.html` where applicable (nav, footer, interaction system, LED CSS)

---

## Making a change end-to-end

1. Edit `index.html` (and `about.html` if needed)
2. Run htmlhint locally to catch markup errors
3. Start local server and run Lighthouse (see [testing.md](testing.md))
4. Commit with a conventional commit message, e.g.:
   - `feat(ui): …` for new visual features
   - `fix(a11y): …` for accessibility fixes
   - `chore: …` for version bumps, config changes
5. Push to `main`
6. Watch CI pass in GitHub Actions — the bot will push `styles.css` if classes changed
7. Confirm the live site at kiroku.io reflects the change (allow ~60s for Pages to rebuild)
